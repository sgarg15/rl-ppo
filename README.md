# PPO Reinforcement Learning

This project trains and evaluates a Proximal Policy Optimization (PPO) agent with PyTorch and Gymnasium.

The implementation supports both discrete and continuous control environments:

- `CartPole-v1`
- `LunarLander-v3`
- `BipedalWalker-v3`

The main training loop is in `train.py`, the actor-critic models are in `ac.py`, evaluation is in `eval.py`, and training plots are generated by `plotting.py`.

## Setup

Create the conda environment:

```bash
conda env create -f environment.yml
conda activate rl
```

If the environment already exists and you changed `environment.yml`, update it with:

```bash
conda env update -f environment.yml --prune
```

## Training

Train on the default environment, `CartPole-v1`:

```bash
python train.py
```

Train on a specific environment:

```bash
python train.py --env LunarLander-v3
python train.py --env BipedalWalker-v3
```

Render the environment while training:

```bash
python train.py --env CartPole-v1 --render
```

After training, the project saves:

- model weights in `model/`
- training configuration in `model/`
- training curves in `figures/`

## Evaluation

Evaluate a trained model:

```bash
python eval.py --env CartPole-v1
```

Run evaluation without rendering:

```bash
python eval.py --env LunarLander-v3 --no-render
```

Evaluate with stochastic actions instead of deterministic actions:

```bash
python eval.py --env BipedalWalker-v3 --stochastic
```

## How It Works

PPO collects a rollout of experience from the environment, estimates advantages with Generalized Advantage Estimation (GAE), and then updates the actor-critic model for several epochs.

```text
Start training
    |
    v
Use the current policy to choose actions
    |
    v
Collect a rollout from the environment
    |
    v
Compute returns and GAE advantages
    |
    v
Compute the PPO loss
    |
    v
Backpropagate the loss
    |
    v
Optimizer updates the model weights
    |
    v
Repeat for many updates
    |
    v
Save the model and training plots
```

In simpler words: PPO tries the current policy, learns which actions were better or worse than expected, updates the model a little bit, and repeats that process many times.

The actor chooses actions. The critic estimates how good a state is. The optimizer updates both networks using a combined loss:

- actor loss: improves the policy
- critic loss: improves value prediction
- entropy bonus: encourages exploration

PPO uses a clipped policy objective so each update improves the policy without changing it too aggressively.

## Environment Presets

Each environment has its own preset in `train.py`:

- learning rate
- rollout steps
- number of updates
- hidden layer size
- entropy coefficient
- reward scale
- value loss coefficient
- optional position penalty

These presets matter because the same hyperparameters do not transfer cleanly between environments. CartPole is short and simple, LunarLander has larger and more varied rewards, and BipedalWalker is a harder continuous-control task.

## Learning Notes

### Reward scaling can make or break training

In LunarLander, rewards can become much larger than in CartPole. When rewards are large, the critic loss can dominate the total PPO loss. If the critic loss becomes too large, the shared gradient clipping can shrink the actor updates, which means the policy may stop improving even though training is still running.

Scaling LunarLander rewards with `reward_scale = 0.1` helped stabilize training by keeping the critic loss closer to the actor loss. The important lesson is that reward magnitude affects optimization, not just logging.

### Environment parameters make a big difference

Hyperparameters that work well in one environment can fail in another. Rollout length, learning rate, entropy bonus, hidden size, and reward shaping all interact with the environment's episode length, reward range, and difficulty.

For example, CartPole can train with shorter rollouts and a larger learning rate. LunarLander needs longer rollouts and more careful reward scaling. BipedalWalker needs a continuous action policy and a larger model because the action space and control problem are more complex.

The lesson: tuning PPO is not only about the algorithm. It is also about the environment.

### GAE is computed in reverse order

Generalized Advantage Estimation (GAE) uses future information to decide how much better or worse an action was than the critic expected. The advantage at one timestep depends on the reward at that timestep, the value of the next state, and the advantage estimate from the next timestep.

Because each advantage depends on the future advantage, the rollout is processed backward. Starting from the end lets the code carry one running `gae` value backward through time. If the calculation went forward, the future advantage would not be known yet.

This is why `train.py` loops through the rollout with `reversed(range(rollout_steps))`. It is working from the last collected transition back to the first one so each timestep can include the discounted effect of what happened after it.

### Discrete and continuous environments need different policies

Discrete environments have a fixed set of actions. In CartPole, the policy chooses between actions like moving left or right. The model outputs action logits, and actions are sampled from a categorical distribution.

Continuous environments require real-valued actions. In BipedalWalker, the policy outputs continuous motor controls. The model outputs means and standard deviations for a normal distribution, samples raw actions, then squashes them with `tanh` so they fit the environment's action bounds.

Because of this, the project has two actor-critic implementations:

- `DiscreteActorCritic` for `gym.spaces.Discrete`
- `ContinuousActorCritic` for `gym.spaces.Box`

The training loop can stay mostly the same because both models expose the same methods: `act`, `evaluate_actions`, and `get_value`.
